    # Meow_Daily 项目技术规格说明书 (Technical Specification)

    ## 1. 项目概述

    **Meow_Daily** 是一款专为多猫家庭设计的无服务器（Serverless）宠物健康与生活记录工具。本项目利用 GitHub 生态系统（Pages, Actions, API, Secrets）结合 PWA（渐进式 Web 应用）技术，提供轻量级、跨平台且完全免费的个人数据追踪解决方案。

    项目深度融合了 **AI 大语言模型能力**（通过 ChatAnywhere API），实现极简的“一句话自然语言录入”、高度拟人化的宠物视角日常提醒，以及基于历史数据的每周健康与开销总结；并结合 Bark 推送服务，打造兼具高实用性与极强情绪价值的智能宠物工具。

    ## 2. 技术栈与运行环境

    - **前端展示与宿主环境**：GitHub Pages (静态网页托管)。
    - **应用形态**：PWA (Progressive Web App)，支持添加至手机桌面，提供类原生体验。
    - **数据库引擎**：GitHub Repository (使用单个 JSON 文件作为纯文本数据库)。
    - **数据读写授权**：用户在初次使用时输入 GitHub Personal Access Token (PAT)，后续操作免密。
    - **AI 赋能服务**：ChatAnywhere API (兼容 OpenAI 接口，如 `gpt-3.5-turbo` 或 `gpt-4o-mini`)，用于前端结构化录入解析与后端文本生成。
    - **自动化与定时任务**：GitHub Actions (利用 Cron 触发器执行定时检查、调用 AI 生成报告及消息推送)。
    - **消息通知**：Bark (iOS 端自定义 HTTP 请求推送工具)，通过 GitHub Secrets 安全配置。

    ## 3. 系统架构设计

    系统采用前端+无服务器架构，数据完全掌握在个人 GitHub 仓库中。

    ### 3.1 数据流与“提交同步”机制

    1. **读取与本地初始化**：前端 PWA 启动时，通过 Fetch API 拉取仓库中的 `db.json` 文件，并将其加载到前端的本地状态（Local State）中进行渲染。
    2. **本地暂存与修改**：用户在界面上进行的新增、修改、删除操作，默认只更新前端本地内存中的数据状态，页面 UI 实时响应，但不立即触发网络请求。
    3. **点击提交与云端同步**：页面提供全局的「保存并同步」主按钮。只有用户点击该按钮后，前端才会读取本地缓存的 Token，获取线上文件的最新 `SHA` 值，然后调用 GitHub REST API (`PUT /repos/{owner}/{repo}/contents/{path}`)，将完整的 JSON 数据一次性 Commit 到仓库中。
    4. **AI 定时通知与总结**：GitHub Actions 根据设定的 Cron 表达式运行。读取最新 JSON 数据，触发 ChatAnywhere API 生成文案，随后遍历 Secrets 中的 `BARK_KEY`，逐个向绑定的设备发送推送。

    ## 4. 功能需求详细说明 (PRD)

    ### 4.1 档案管理模块 (Profile)

    支持多只猫咪的档案独立管理。

    - **基础字段**：猫咪 ID、姓名、性别、出生日期、到家日期。
    - **业务逻辑**：前端自动计算并展示当前年龄和陪伴天数。

    ### 4.2 记录管理模块 (Records) - 增删改查与时间选择

    支持对日常记录的精细化管理。

    - **时间选择功能**：所有记录表单必须包含时间选择器（Date-Time Picker），默认填入当前时间，但允许用户精确回溯修改具体的日期与具体时间（精确到分钟）。
    - **编辑与删除功能 (CRUD)**：每条历史记录卡片右侧提供「编辑」和「删除」操作项。编辑可唤起表单重填，删除需弹窗二次确认。修改均在本地暂存，点击主按钮后同步至云端。
    - **数据分类标准**：
      - **日期记录**：事件类型、精确时间、备注。
      - **猫粮记录**：品牌、种类、单价、每日食量、起止时间。
      - **体重记录**：精确称重时间、体重数值（kg）。
      - **就诊记录**：时间、医院、病因、检查、过程、结果、花费及长备注。

    ### 4.3 AI 智能录入增强

    支持手动表单与 AI 语音/文本双轨录入。

    - **一句话记录**：用户输入“昨天晚上8点给汤圆称了体重，5.2kg”。
    - **解析逻辑**：前端请求 ChatAnywhere API，利用 `json_object` 格式强制 AI 返回包含时间戳、记录类型和数据的结构化信息，前端解析后自动生成一条暂存记录，等待用户点击同步。

    ### 4.4 提醒与通知模块 (Reminders)

    依托 GitHub Actions、ChatAnywhere 和 Bark 实现多设备触达。

    - **AI 生成日常提醒**：触发条件满足时（如换猫砂周期到达），Action 脚本调用 AI 生成宠物视家的催促文案（例如：“铲屎的！我的厕所已经 14 天没大扫除了，快去换猫砂！😾”）。
    - **AI 每周总结 (Weekly Report)**：每周日触发 Cron 任务，读取近 7 天数据，调用 AI 生成精炼的周度健康与开销总结，推送到所有绑定的 Bark 设备。

    ## 6. AI 接口调用规范 (ChatAnywhere API)

    在项目中调用 AI 进行“一句话智能记事”时，为了防止 AI 产生幻觉并确保数据格式绝对稳定，必须采用严格的 `System Prompt` 约束，且**禁止 AI 生成绝对时间戳，时间戳和唯一 ID 将由前端代码在接收到解析数据后统一生成并注入**。

    - **API Base URL**: `https://api.chatanywhere.tech`
    - **接口路径**: `/v1/chat/completions`
    - **请求方式**: `POST`
    - **鉴权方式**: Header 添加 `Authorization: Bearer {API_KEY}`

    ### 6.1 严格约束的请求体示例 (Request Body)

    JSON

    ```
    {
      "model": "gpt-3.5-turbo",
      "response_format": { "type": "json_object" },
      "messages": [
        {
          "role": "system",
          "content": "你是一个严格的宠物日记数据提取API。请将用户的自然语言转化为精确的JSON格式。不要生成任何绝对时间戳（时间戳由前端系统自动生成）。\n必须严格遵守以下JSON结构返回，缺失的数据用null表示：\n{\n  \"category\": \"必须是以下枚举值之一：routine(日常护理), food(饮食), weight(体重), medical(医疗)\",\n  \"parsed_data\": {\n    // 若 category=weight，必须包含: \"weight_kg\"(数字), \"note\"(字符串)\n    // 若 category=routine，必须包含: \"type\"(如驱虫、洗澡、换猫砂等), \"note\"(字符串)\n    // 若 category=food，必须包含: \"brand\"(品牌), \"type\"(干粮/罐头等), \"daily_intake_g\"(数字)\n    // 若 category=medical，必须包含: \"hospital\"(医院), \"symptom\"(症状), \"treatment\"(治疗方案), \"cost\"(数字金额)\n  },\n  \"mentioned_time\": \"提取用户话语中提及的时间状语（如'昨天晚上8点'、'刚刚'），若未提及则返回空字符串。\"\n}\n严禁输出任何多余的解释性纯文本。"
        },
        {
          "role": "user",
          "content": "汤圆刚才拉软便了，带去瑞鹏医院看了下，医生开了点益生菌，总共花了150块。"
        }
      ]
    }
    ```

    ### 6.2 预期的标准响应体示例 (Response)

    前端接收到该 JSON 后，即可极其方便地进行反序列化处理：

    JSON

    ```
    {
      "id": "chatcmpl-xxxxx",
      "choices": [
        {
          "message": {
            "role": "assistant",
            "content": "{\n  \"category\": \"medical\",\n  \"parsed_data\": {\n    \"hospital\": \"瑞鹏医院\",\n    \"symptom\": \"软便\",\n    \"treatment\": \"开益生菌\",\n    \"cost\": 150,\n    \"note\": \"\"\n  },\n  \"mentioned_time\": \"刚才\"\n}"
          }
        }
      ]
    }
    ```

    ### 6.3 前端代码侧的处理逻辑 (补充约束)

    当前端 Fetch 到上述 `content` 字符串并解析为 JSON 对象后，必须执行以下代码级组装，才能存入本地状态并同步至云端：

    1. **生成 ID**：通过 `Date.now() + Math.random()` 为记录生成唯一的 `record_id`。
    2. **时间戳注入**：如果 AI 提取的 `mentioned_time` 为空 or “刚才”，则获取系统当前时间格式化为 `YYYY-MM-DD HH:mm` 写入 `timestamp` 字段。如果识别到具体语境（如“昨天”），前端则通过 JS 的 `Date` 对象进行相应的时间回溯计算。
    3. **推入对应的数组**：根据返回的 `category` 字段，将组合好的对象 `push` 到对应猫咪的 `records[category]` 数组中，并渲染到 UI 上等待用户点击「保存并同步至云端」。
