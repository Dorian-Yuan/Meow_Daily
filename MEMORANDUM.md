# Meow_Daily 开发核心准则 (Memorandum)

## ⚠️ 绝对禁令 (Hard Rules)
1. **规格说明书不可篡改**：`Meow_Daily 项目技术规格说明书 (Technical Specification).md` 是项目的圣经，**禁止任何形式的修改**（包括修正错误或更新示例）。它必须保持原始状态作为只读参考。
2. **严禁先斩后奏**：如果规格说明书中的描述不够清晰、存在歧义或涉及体验决策，**必须先通过 `ask_user` 请教**，获得明确答复后方可实施。
3. **撤回即禁区**：一旦用户要求撤回或表达不满的操作，严禁再次触碰。

## 🐈 角色与命名规范
1. **首席 IP 形象**：橘猫 **“岁岁”**。
2. **示例规范**：在代码逻辑、UI 渲染、AI 提示词、本地数据库示例中，统一使用“岁岁”作为默认猫咪名称。

## 🛠️ 工作流程
1. **前置检查**：每次开始新任务前，必须阅读此 `MEMORANDUM.md`。
2. **版本号强制更新与 Git 提交**：每次涉及逻辑变更、修复或 UI 调整的更新，**必须手动递增版本号**（语义化版本，如 V2.0.1 -> V2.0.2），并确保其在“设置”页底部及控制台日志中正确显示。修改完成后，必须立即通过 `git commit` 提交修改，并 `git push` 上传到远程仓库。
3. **页面职能分工**：
   - **首页 (Home)**：仪表盘、快速录入入口（日常、饮食、体重、就诊）、最新状态概览。
   - **记录 (Records)**：按月聚合的历史流水、CRUD（编辑/删除）管理。
   - **我的 (Profile)**：岁岁档案（含重要里程碑）、系统设置入口。
3. **按钮命名**：统一使用两个字：日常、饮食、体重、就诊。
4. **进度实时化**：每完成一个原子任务，必须在 `task.md` 中打勾。
5. **资源管理**：
   - 图像资源（PNG/JPG）由用户提供。
   - 需要新资源时，需明确告知用户所需规格，由用户传输。

## 🎨 UI/UX 精神

- 品牌蓝 (#4066E0) + 岁岁橘 (#F28C38)。
- 极致圆润、扁平治愈系。

### 项目执行框架：

1. 需求探索
- 通过提问来理解我真正需要的东西，而不只是我说的
- 如果我的假设不合理，大胆质疑
- 帮我区分「现在必须有」和「以后再加」
- 如果想法太大，建议一个更聪明的切入点

2. 方案规划
- 明确提出第一版（V1）要构建的内容
- 用通俗易懂的语言解释技术方案
- 评估复杂度：简单 / 中等 / 有挑战
- 列出我需要准备的东西（账号、服务、决策）
- 展示成品的大致轮廓

3. 分步构建
- 分阶段构建，让我看到进展并给出反馈
- 边做边讲解（我想学习）
- 每一步都经过测试再继续
- 在关键决策点停下来征求我的意见
- 遇到问题时，给我选项而不是直接替我决定

4. 精细打磨
- 让它看起来专业，不像一个黑客马拉松作品
- 优雅地处理边界情况和错误
- 确保速度快，在不同设备上正常运行
- 添加让产品有「完成感」的小细节

5. 交付上线
- 如果需要上线，帮我部署
- 提供清晰的使用、维护和修改说明
- 完整文档化，让我不依赖于本次对话
- 告诉我 V2 版本可以添加或改进的内容

6. 协作守则
- 把我当作产品负责人 —— 我做决策，你来实现
- 不要用技术术语淹没我，用人话翻译一切
- 如果我在把事情复杂化或走弯路，请直接推回
- 对局限性保持坦诚 —— 我宁愿调整预期，也不想失望
- 快速推进，但不要快到我跟不上节奏

### 底线原则

- 我不只是要它能用 —— 我要它是我骄傲展示给别人看的东西
- 这是真的。不是原型，不是 Demo，是一个可运行的产品
- 让我始终掌控全局，始终了解进展
